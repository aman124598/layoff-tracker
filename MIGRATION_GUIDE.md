# Database Migration Guide

## Overview
This guide helps you update your existing Supabase database to support the new multi-source layoff tracking feature.

## What's Changing
The `layoffs` table is being enhanced to track the source of each layoff record.

### Changes Made
1. **New Column**: `source_name` - TEXT field to store the source (NewsAPI, Layoffs.fyi, etc.)
2. **New Column**: `updated_at` - Timestamp for tracking last update
3. **New Index**: On `source_name` for faster filtering

## Migration Steps

### Option 1: Using Supabase Web UI (Recommended)

1. **Login to Supabase**
   - Go to https://supabase.com
   - Select your project

2. **Open SQL Editor**
   - Click "SQL Editor" in the left sidebar
   - Click "New Query"

3. **Run Migration Script**
   ```sql
   -- Add source_name column
   ALTER TABLE layoffs ADD COLUMN source_name TEXT DEFAULT 'Unknown' NOT NULL;
   
   -- Add updated_at column
   ALTER TABLE layoffs ADD COLUMN updated_at TIMESTAMP WITH TIME ZONE DEFAULT TIMEZONE('utc'::text, NOW()) NOT NULL;
   
   -- Create index for faster source filtering
   CREATE INDEX IF NOT EXISTS idx_layoffs_source ON layoffs(source_name);
   
   -- Update existing records to have a default source
   UPDATE layoffs SET source_name = 'Unknown' WHERE source_name IS NULL;
   ```

4. **Verify**
   - Query the table: `SELECT * FROM layoffs LIMIT 1;`
   - Should see `source_name` column with "Unknown" value

### Option 2: Using psql CLI

```bash
psql postgresql://[user]:[password]@[host]:5432/[database] << EOF
ALTER TABLE layoffs ADD COLUMN source_name TEXT DEFAULT 'Unknown' NOT NULL;
ALTER TABLE layoffs ADD COLUMN updated_at TIMESTAMP WITH TIME ZONE DEFAULT TIMEZONE('utc'::text, NOW()) NOT NULL;
CREATE INDEX IF NOT EXISTS idx_layoffs_source ON layoffs(source_name);
UPDATE layoffs SET source_name = 'Unknown' WHERE source_name IS NULL;
EOF
```

### Option 3: Update Schema File & Redeploy

If you want to recreate the entire table with the new schema:

1. **Backup your data** (IMPORTANT!)
   ```sql
   CREATE TABLE layoffs_backup AS SELECT * FROM layoffs;
   ```

2. **Drop the old table**
   ```sql
   DROP TABLE layoffs;
   ```

3. **Create new table** with updated schema:
   ```sql
   CREATE TABLE IF NOT EXISTS layoffs (
       id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
       company_name TEXT NOT NULL,
       layoff_date DATE NOT NULL,
       employees_laid_off INTEGER,
       country TEXT NOT NULL,
       industry TEXT NOT NULL,
       source_url TEXT NOT NULL UNIQUE,
       source_name TEXT DEFAULT 'Unknown' NOT NULL,
       created_at TIMESTAMP WITH TIME ZONE DEFAULT TIMEZONE('utc'::text, NOW()) NOT NULL,
       updated_at TIMESTAMP WITH TIME ZONE DEFAULT TIMEZONE('utc'::text, NOW()) NOT NULL
   );
   ```

4. **Create indexes**
   ```sql
   CREATE INDEX IF NOT EXISTS idx_layoffs_company_name ON layoffs(company_name);
   CREATE INDEX IF NOT EXISTS idx_layoffs_date ON layoffs(layoff_date);
   CREATE INDEX IF NOT EXISTS idx_layoffs_country ON layoffs(country);
   CREATE INDEX IF NOT EXISTS idx_layoffs_industry ON layoffs(industry);
   CREATE INDEX IF NOT EXISTS idx_layoffs_source ON layoffs(source_name);
   ```

5. **Restore data** (if doing full migration)
   ```sql
   INSERT INTO layoffs (company_name, layoff_date, employees_laid_off, country, industry, source_url, source_name, created_at)
   SELECT company_name, layoff_date, employees_laid_off, country, industry, source_url, 'Imported' as source_name, created_at
   FROM layoffs_backup;
   ```

## Verification Checklist

After migration, verify:

- [ ] New `source_name` column exists
- [ ] New `updated_at` column exists  
- [ ] All existing records have `source_name` value (shouldn't be NULL)
- [ ] Index `idx_layoffs_source` is created
- [ ] Can query by source: `SELECT * FROM layoffs WHERE source_name = 'Unknown';`
- [ ] Total record count matches before migration

## Rollback (If Needed)

If something goes wrong:

```sql
-- Drop the new columns if you want to revert
ALTER TABLE layoffs DROP COLUMN source_name;
ALTER TABLE layoffs DROP COLUMN updated_at;
DROP INDEX idx_layoffs_source;
```

## Common Issues

### Issue: "Column already exists"
**Solution**: The column already exists. Skip the ADD COLUMN statement.

### Issue: "Constraint violation"
**Solution**: Ensure `source_name` is NOT NULL and has a DEFAULT value.

### Issue: "Index already exists"
**Solution**: Use `CREATE INDEX IF NOT EXISTS` (already included above).

### Issue: Data loss after migration
**Solution**: You should have created a backup table first. Restore from `layoffs_backup` table.

## Performance Impact

- **Minimal**: New columns are simple TEXT and TIMESTAMP fields
- **Index creation**: May take 1-2 seconds on large tables (1000+ records)
- **Query performance**: Will be improved by the new index on `source_name`
